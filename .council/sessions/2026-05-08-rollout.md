# Council session: Quiet Ledger rollout + structure + weekly scrape

## Why convened

The Chairman has approved Quiet Ledger and opened the Council. The
PR (#17) lands the architectural scaffold, the spike, and the
foundational design tokens. Three streams of follow-on work are now
sequenced.

## Scope

Three phases, in order. Each phase is its own branch and PR.

### Phase A — Quiet Ledger production application

Apply Quiet Ledger to the rest of the production site. The
foundational `.dp-*` retune already shipped on PR #17, so card
review pages, bank hubs, airline programme pages, salary-transfer
slug pages, and valuations inherit the new palette and component
rules for free. The remaining work:

- **`src/pages/index.astro` — full rewrite** to the 4-section spec
  (masthead-dateline + 5-story typographic index → live module
  combining tracker + deals → latest analysis feed → newsletter +
  RSS-by-band). Replaces ~668 lines of scoped CSS with `.dp-*`
  classes and component imports. The hardest single change in this
  phase; requires browser preview at 390px and 1280px.
- **`src/pages/index.astro` — kill the deal-rail negative-margin
  bug** by collapsing the rail into the live module. Resolves the
  `index.astro:163-177` paper cut documented in the audit.
- **`src/components/Header.astro` — review** the desktop padding
  scale once the homepage rebuild lands; if `.wrap` standardises
  the 16/24/32 inline scale, header padding inherits it. Otherwise
  no further header changes are needed.
- **Remove `Header.astro` `.wrap` overrides** if the homepage
  rebuild establishes a single `.wrap` rule consumable site-wide.
- **Spot-check `/design-spike/`** under the new global tokens. The
  spike file scopes its own tokens (`--ql-*`) so should be
  unaffected; if any drift appears, reconcile.

**Owner:** technical-lead (build + deploy), section editors do
copy review. Chairman gate at PR review.

**Acceptance:**
- Lighthouse mobile ≥ 95 on the rebuilt homepage.
- `astro check` clean; `npm run build` succeeds.
- 8 sections become 4 visibly; deal rail merges with tracker; gold
  appears nowhere on the homepage.
- All long-form pages (card review, guide, bank hub, airline
  programme, valuations) still render cleanly under the retuned
  `.dp-*` tokens.

**Branch:** `claude/quiet-ledger-rollout`.

### Phase C — Seed-then-watch Firecrawl cadence (the data flywheel)

The whole point of the structure-and-design phase: once governance,
structure, and look/feel are up, the Council fills the data layer
via Firecrawl-driven scrapes of bank websites — **initial bulk seed
across the priority banks, then weekly cron picks up deltas.** The
cadence is not "one bank a fortnight"; it is "all priority banks
seeded as fast as scrapers can ship, then the existing weekly cron
handles change detection from there."

**Step status as of 2026-05-08:**

1. ~~**Restore `welcomeBonus` to `SCRAPED_FIELDS`.**~~ **Done.**
   Chairman-approved 2026-05-08; landed with contract tests on
   PR #17. STATUS.md Q-A resolved.

**Remaining steps in order:**

2. **Per-bank perk-filter regex** — generalise the FAB-specific
   filter at `_normaliser.ts:130` to a per-bank override map keyed
   by `bankSlug`. ENBD will need its own regex for the
   `emiratesnbd.com`, `online.emiratesnbd.com` chrome. ADCB will
   need its own for `adcb.com`. Pattern is reusable from there.
3. **Bank registry** — `scripts/scrape/banks.registry.json`
   listing every implemented scraper. New banks get added by
   appending to the registry, not by editing the workflow.
4. **Workflow matrixisation.** Replace the hard-coded `npm run
   scrape:fab` line in `.github/workflows/scrape.yml` with a matrix
   job iterating bank slugs from the registry. `fail-fast: false`
   so one bank's failure does not abort the others. Each matrix job
   commits its own scrape JSON; a single follow-up job runs
   `npm run scrape:propose` against all banks before opening one
   weekly PR.
5. **Per-bank scraper rollout (priority order).** The user has
   asked for the data layer seeded across all priority banks,
   not paced one per fortnight. The plan: ship the per-bank
   scrapers in this order, each as its own bite-sized PR with
   Head-of-Research dossier first, scraper second, fixture and
   tests third:

   1. **Emirates NBD** — 9 pre-seeded cards in `cards.json`; this
      scraper unlocks the most existing data. Highest-priority.
   2. **ADCB** — Touchpoints earn structure overlaps strongly with
      ENBD; tooling reuse high. Salary-transfer programme dense.
   3. **Mashreq** — Mashreq Salaam currency + noon co-brand;
      cashback-card-heavy beat for Business & Real Estate Editor.
   4. **HSBC UAE** — global brand, simpler page chrome, easier
      scrape; good for catching tooling regressions.
   5. **RAKBank** — high salary-transfer offer cadence; tracker
      relevance high.
   6. **Emirates Islamic** — Islamic-card scrape; tests Sharia-
      compliant card structure end-to-end.
   7. **CBD** — Super Saver and lifestyle cards.
   8. **DIB** — second Islamic-card scrape; DIB and Emirates
      Islamic together cover the Sharia-compliant lineup.
   9. **Citi UAE** — international brand, simpler scrape.
   10. **Standard Chartered UAE** — international brand.
   11. **ADIB** — third Islamic-card scrape, completing the lineup.

6. **Initial seed run.** Once all priority bank scrapers are
   implemented, kick off an out-of-cycle scrape via the
   `workflow_dispatch` trigger to seed `cards.json` across the full
   priority list in a single PR. This is the "initial scrape"
   before the weekly cron takes over.
7. **Weekly cron handles change detection from there.** The
   existing Sunday 23:00 UTC cron runs the matrix job; opens a PR
   per week if any bank's `cards.json` slice changes; never
   auto-merges; Managing Editor reviews with the Fact-Checker on
   standby. No human intervention required for steady-state.

**Branches:** one per bank for steps 5.x (e.g.
`claude/scrape-enbd`, `claude/scrape-adcb`). Steps 2 (perk regex),
3 (registry), and 4 (workflow matrix) land in their own PRs first.
